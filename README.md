# Supervised Machine Learning Pipeline for Logistics Risk Prediction

Predict delivery delays before they happen using a supervised classification model, enabling proactive supply chain intervention.

**Business problem:** Delayed shipments drive inventory holding costs and erode customer loyalty. A false negative (missing a predicted delay) is far more costly than a false positive, so the model is tuned to favor recall over precision.

## Project Structure

```
├── app/
│   ├── api.py                # FastAPI backend — loads model once, serves /predict + /health
│   └── streamlit_app.py      # Streamlit frontend — form UI, calls the FastAPI backend over HTTP
├── data/
│   ├── raw/                  # Immutable source data (downloaded, gitignored)
│   └── processed/            # Cleaned, split datasets (parquet, gitignored)
├── models/                   # Serialized model artifacts (generated by training, gitignored)
├── src/
│   ├── config.py             # Central configuration & hyperparameters
│   ├── features.py           # Feature engineering transforms
│   ├── ingest.py              # Data download & caching
│   ├── preprocess.py         # Cleaning, encoding, splitting
│   ├── predictor.py           # Predictor class — single inference entry point (api.py, evaluate.py)
│   ├── train.py               # Training pipeline & threshold tuning
│   ├── evaluate.py            # Evaluation metrics, feature importance, leakage audit
│   └── log_utils.py          # Centralized logging configuration
├── tests/
│   ├── test_preprocess.py
│   ├── test_features.py
│   └── test_integration.py   # End-to-end: synthetic data → train → predict → API
├── .github/workflows/ci.yml  # Lint (ruff), type check (mypy), tests, Docker build
├── Dockerfile                # Multi-stage build: trains the model, then packages api + streamlit
├── run.sh                    # Container entrypoint: runs FastAPI (8000) + Streamlit (8501)
├── requirements.txt
├── IMPLEMENTATION-PLAN.md
└── README.md
```

**Architecture:** the Streamlit UI never loads the model directly — it calls a FastAPI backend
over HTTP (`POST /predict`). The backend loads the model once at startup and holds it in memory.
This keeps the model-serving code path identical whether it's called from the UI, `curl`, or
another service, and lets the two processes scale/restart independently. Both the API and the
offline evaluation script (`src/evaluate.py`) go through the same `src/predictor.py::Predictor`
class, so there is exactly one inference code path to keep correct.

## Quickstart

```bash
# 1. Install dependencies
pip install -r requirements.txt

# 2. Train the model — downloads + caches the dataset, preprocesses, engineers
#    features, tunes hyperparameters + decision threshold, and writes all
#    artifacts to models/
python -m src.train

# 3. Launch the backend API (loads the trained model once)
python -m uvicorn app.api:app --host 0.0.0.0 --port 8000 &

# 4. Launch the dashboard (talks to the API above)
#    Set API_BASE_URL if the backend isn't on localhost:8000.
streamlit run app/streamlit_app.py
```

Or via Docker, which does all of the above in one container:

```bash
docker build -t logistics-risk .
docker run -p 8501:8501 -p 8000:8000 logistics-risk
# UI:  http://localhost:8501
# API: http://localhost:8000/docs
```

## Dataset

[DataCo Smart Supply Chain for Big Data Analysis](https://www.kaggle.com/datasets/shashwatwork/dataco-smart-supply-chain-for-big-data-analysis) — 180K+ records merging sales, logistics, and supply chain topology.

**Key characteristics:**
- ~55/45 class balance (late vs on-time)
- `Shipping Mode` and the historical per-mode delay rate are the dominant predictors
- Columns that encode post-delivery/post-shipment knowledge (`Delivery Status`, `Days for shipping (real)`,
  `Order Status`, `Benefit per order`, `Sales per customer`, and the actual `shipping date (DateOrders)`
  itself) are excluded from training — see [Key Design Decisions](#key-design-decisions).

---

## Model

- **Algorithm:** XGBoost with `RandomizedSearchCV` + `StratifiedKFold(5)`
- **Hyperparameter tuning:** 30 iterations over learning_rate, max_depth, subsample, colsample_bytree, reg_lambda, min_child_weight
- **Threshold tuning:** Custom decision boundary tuned via F2-score optimization (recall-weighted)
- **Evaluation:** ROC-AUC, Precision-Recall, F2-score, confusion matrix, leakage audit
- **Serialization:** XGBoost native `save_model()` (UBJSON format) — avoids pickle RCE risks

**Current held-out test performance:** ROC-AUC ≈ 0.74, F2 ≈ 0.86 (recall ≈ 1.00, precision ≈ 0.56
at the F2-optimal threshold of 0.15). Exact numbers change slightly between reruns since the
model is retrained from a fresh hyperparameter search — see `models/experiments.csv` after
training for the latest run.

**Known limitation:** because F2 weights recall 4× over precision and false negatives are the
costlier error for this business, the auto-tuned threshold currently lands very low, close to
"flag nearly every shipment as at-risk." That maximizes the F2 metric but has limited standalone
operational value — a team using this in practice should treat the output probability as a
*ranking* signal (triage the highest-probability shipments first) rather than acting on the
binary label alone, or pick a stricter, business-chosen threshold (see the precision/recall
trade-off table logged by `src/evaluate.py` / `models/experiments.csv`).

## Deployment

FastAPI backend (`app/api.py`) + Streamlit frontend (`app/streamlit_app.py`), run together via
`run.sh` / the Docker image:
- `POST /predict` — single-shipment risk score + SHAP-style feature contributions
- `GET /health` — readiness probe (reports `starting` until the model finishes loading)
- Form inputs for shipment parameters, customer location, shipping mode
- Real-time probability + risk classification (HIGH/LOW)
- Decision threshold display
- Expandable prediction details and feature-contribution chart

## Key Design Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Model | XGBoost | Best AUC on DataCo benchmarks, clean sklearn integration, compact serialization |
| Serialization | `.ubj` (native format) | Avoids pickle RCE risk + cross-version compatibility |
| Deployment | Streamlit | Ops staff need form-based UI, not APIs |
| Threshold | F2-optimized (recall-weighted) | Recall-weighted; FN >> FP in business context (see Known Limitation above) |
| CV strategy | StratifiedKFold(5) + chronological holdout | Handles class balance + temporal structure |
| Leakage prevention | Strict exclusion of post-hoc columns | `Delivery Status`, `Days for shipping (real)`, `Order Status`, `Benefit per order`, `Sales per customer` are dropped, and the actual `shipping date (DateOrders)` is never used to build features — that column only exists once a shipment is complete, so any feature derived from it (e.g. a naive "processing days" = ship date − order date) reconstructs the leaked "real shipping days" column and inflates offline metrics without being available at real prediction time. Only `order date (DateOrders)` (known when the order is placed) is used for temporal features. |

## Testing & CI

`.github/workflows/ci.yml` runs on every push/PR to `main`: `ruff` lint, `mypy` strict type check,
unit tests, the full integration test (trains a small model on synthetic data end-to-end and hits
